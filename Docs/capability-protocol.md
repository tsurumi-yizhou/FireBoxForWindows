# FireBox Capability Protocol

## 1. Scope

This document defines the capability-plane IPC contract for FireBox. It is used by trusted client applications to perform AI operations: chat completion, streaming chat, embeddings, and structured function calls.

**The caller works with model IDs directly** (e.g., `gpt-4.1`, `claude-sonnet-4-20250514`, `text-embedding-3-large`). The caller specifies which model to use. The service handles provider selection, routing, and failover — none of which are exposed to the caller.

This document includes the `ChatStreamSink` event interface for streaming chat.

## 2. General Conventions

### 2.1 Result Envelope

All synchronous AI methods return a tagged result envelope:

```text
Result<T> {
  response?: T
  error?: string
}
```

**Invariant:** Exactly one of `response` or `error` MUST be present.

When `error` is present, it is a human-readable error message string. The service SHOULD pass through the upstream provider's error message directly when the failure originates from a provider. For service-internal failures (bad arguments, no available provider for the requested model, etc.), the service produces its own descriptive message.

There are no structured error codes.

### 2.2 Streaming Model

Streaming chat uses a "setup response + event stream + terminal event" pattern:
1. `StartChatCompletionStream` returns `requestId` immediately
2. Events are delivered via `ChatStreamSink` callbacks
3. Exactly one terminal event ends the stream: `Completed`, `Error`, or `Cancelled`
4. No events are sent after a terminal event

### 2.3 Request ID

`requestId` is a 64-bit integer that correlates a streaming chat request with its event stream. Generated by the service.

### 2.4 Authentication and Authorization

All methods require the caller to be authorized.

Platform-specific identity mechanisms:
- **Android Binder**: UID + package name + signing identity
- **Windows COM**: PID + process name + executable path (allowlist)
- **macOS XPC**: Audit token / code signing identity

Unauthorized calls result in a transport-level rejection or an error string.

### 2.5 Candidate Failover

The service MAY internally try multiple providers that offer the same `modelId`. This is invisible to the caller — the caller receives only the final result or error.

```text
ModelInfo {
  modelId: string

```

### 4.3 ModelCapabilities

```text
ModelCapabilities {
  reasoning: bool

  inputFormats: MediaFormat[]
  outputFormats: MediaFormat[]
}
```

### 4.4 ChatMessage

```text
ChatMessage {
  role: string                   // "system", "user", or "assistant"
  content: string
  attachments: ChatAttachment[] = []
}
```

### 4.5 ChatAttachment

```text
ChatAttachment {
  mediaFormat: MediaFormat
  mimeType: string
  fileName?: string
  data: binary                   // transport-specific binary handle
  sizeBytes: int64 = -1         // -1 means unknown
}
```

Transport mapping for `data`:
- **Android Binder**: `ParcelFileDescriptor`
- **Windows COM**: Byte array or binary stream
- **macOS XPC**: `NSData` or file handle

```text
ChatCompletionRequest {
```

## 5. CapabilityService Methods

### 5.1 `Ping`

| | |
|---|---|
| **Request** | `{ message: string }` |
| **Response** | `{ result: string }` |
| **Notes** | Connectivity probe. Response is `"Pong: {message}"`. |

### 5.2 `ListModels`

| | |
|---|---|
| **Request** | none |
| **Response** | `{ models: ModelInfo[] }` |
| **Notes** | Returns all models available to the caller, with their capabilities and availability status. No provider or routing details are exposed. |

### 5.3 `ChatCompletion`

| | |
|---|---|
| **Request** | `ChatCompletionRequest` |
| **Response** | `ChatCompletionResult` |
| **Notes** | Synchronous chat completion. |

### 5.4 `StartChatCompletionStream`

| | |
|---|---|
| **Request** | `ChatCompletionRequest` + `ChatStreamSink` reference |
| **Response** | `{ requestId: int64 }` |
| **Notes** | Starts a streaming chat request. All payload data is delivered through `ChatStreamSink`. |

### 5.5 `CancelChatCompletion`

| | |
|---|---|
| **Request** | `{ requestId: int64 }` |
| **Response** | none |
| **Notes** | Best-effort cancellation. Normally produces a terminal `Cancelled` event on the stream. |

### 5.6 `CreateEmbeddings`

| | |
|---|---|
| **Request** | `EmbeddingRequest` |
| **Response** | `EmbeddingResult` |
| **Notes** | Synchronous embedding generation. Returns all embeddings in a single response. |

### 5.7 `CallFunction`

| | |
|---|---|
| **Request** | `FunctionCallRequest` |
| **Response** | `FunctionCallResult` |
| **Notes** | Synchronous structured function call. |

## 6. ChatStreamSink

`ChatStreamSink` is a client-implemented, server-invoked one-way event interface for streaming chat delivery.

Every event carries a `requestId` matching the ID returned by `StartChatCompletionStream`.

### 6.1 `OnStarted`

```text
{
  requestId: int64
}
```

Streaming has started.

### 6.2 `OnDelta`

```text
{
  requestId: int64
  deltaText: string
}
```

Text delta chunk.

### 6.3 `OnReasoningDelta`

```text
{
  requestId: int64
  reasoningText: string
}
```

Reasoning text delta chunk.

### 6.4 `OnUsage`

```text
{
  requestId: int64
  usage: Usage
}
```

Token accounting update. Emitted only when token counts are available and non-zero.

### 6.5 `OnCompleted`

```text
{
  requestId: int64
  modelId: string
  message: ChatMessage            // role is always "assistant"
  reasoningText?: string
  finishReason: string
  usage: Usage
}
```

Final successful response. Terminal event.

### 6.6 `OnError`

```text
{
  requestId: int64
  error: string
}
```

Terminal failure event. The error string is passed through directly from the provider when applicable.

### 6.7 `OnCancelled`

```text
{
  requestId: int64
}
```

Terminal cancellation event.

### 6.8 Event Sequencing Rules

**Typical success sequence:**
```text
StartChatCompletionStream(request) → { requestId }
  OnStarted
  OnDelta*
  OnReasoningDelta*
  OnUsage?
  OnCompleted
```

**Failure sequence:**
```text
StartChatCompletionStream(request) → { requestId }
  [OnStarted?]
  OnError
```

**Cancellation sequence:**
```text
StartChatCompletionStream(request) → { requestId }
  [OnStarted?]
  CancelChatCompletion(requestId)
  OnCancelled
```

**Rules:**
- `OnStarted` SHOULD be emitted before content deltas. Early failures (before provider selection) MAY skip `OnStarted` and go directly to `OnError`.
- `OnDelta` and `OnReasoningDelta` may appear zero or more times, in any interleaving order.
- `OnUsage` may appear zero or more times during streaming.
- Exactly one terminal event ends the stream: `OnCompleted`, `OnError`, or `OnCancelled`.
- No events are sent after a terminal event.

### 6.9 Alternative: Single-Dispatch Model

For platforms where separate callback methods are impractical, implementations MAY use a single-dispatch model:

```text
ChatStreamEvent {
  requestId: int64
  type: int32              // 0=Started, 1=Delta, 2=Usage, 3=Completed, 4=Error, 5=Cancelled, 6=ReasoningDelta
  deltaText?: string
  reasoningText?: string
  usage?: Usage
  modelId?: string
  message?: ChatMessage
  finishReason?: string
  error?: string
}
```

Both dispatch models are semantically equivalent.

## 7. Validation Constraints

Violations result in an error string describing the problem.

### 7.1 Chat Request

- `modelId` MUST be non-empty
- `messages` MUST be non-empty
- Each `message.role` MUST be one of: `system`, `user`, `assistant`
- Each attachment MUST have a non-empty `mimeType`
- Each attachment is limited to **8 MiB**

### 7.2 Embedding Request

- `modelId` MUST be non-empty
- `input` MUST be non-empty
- Total character count across all `input` items MUST NOT exceed **200,000**

### 7.3 Function Call Request

- `modelId` MUST be non-empty
- `functionName` MUST be non-empty
- `inputJson` MUST be non-empty
- `inputSchemaJson` MUST be non-empty
- `outputSchemaJson` MUST be non-empty
